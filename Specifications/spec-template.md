## 2. 概要

### 2.1 背景

- 自分自身が使いやすいと思う論文管理アプリを作りたい
- Zoteroが不便
- .bibを使用するための論文データ出力が物理学の論文向けになっていない。余計な情報が多く、エラーも出やすい。

### 2.2 目的

- チェックボックスでファイルを選択し、まとめて.bib用のテンプレを出力
- .md等で作成したノートを論文と紐付けて管理したい (obsidian的な)
- Zoteroで実装されている論文ファイルをリネームして、フォルダ分けする機能も欲しい
- Cloudやドライブで複数デバイス共有を前提に設計

### 2.3 対象プラットフォーム

- デスクトップアプリ
- 想定 OS: macOS
- 使用技術: Tauri / フロントエンド未定 / バックエンド未定

## 3. 想定ユーザー

### 3.1 メインユーザー

- 研究者
- 大学院生
- 論文を継続的に読むエンジニア

### 3.2 ユーザー課題

- 論文 PDF が散在する
- メタデータ管理が煩雑
- タグやメモの整理が難しい
- あとで読み返す論文を見失う
- 論文執筆の際に引用する.bib作成が面倒

## 4. スコープ

### 4.1 対象範囲

- 上記目的の機能

### 4.2 対象外

- アプリ内でpdfを開く機能

## 6. 機能要件

### 6.1 論文登録

- PDF ファイルを選択して登録できる
- 手動でタイトル、著者、出版年、DOI を入力できる
- DOI からメタデータを取得できる
- 可能な範囲で PDF や DOI からメタデータを自動補完する
- DOI、タイトル、ファイルパスなどを用いて重複登録を検知できる

### 6.2 メタデータ管理

- タイトル
- 著者
- 出版年
- 学会誌 / ジャーナル名
- DOI / arXiv ID
- URL
- タグ
- 評価
- 読書状態
- BibTeX 出力用のキー
- ローカルの PDF パス
- 紐付けノートのパス

### 6.3 検索・絞り込み

- キーワード検索
- 著者検索
- タグ絞り込み
- 読書状態絞り込み
- 年別絞り込み
- ソート条件
- 一覧上で複数論文をチェックボックス選択できる

### 6.4 メモ・ノート

- 論文単位でノートを紐付けて管理できる
- Markdown ファイルを論文に関連付けできる
- 新規ノートファイルを作成して論文に紐付けできる
- ノートファイルを外部エディタで開ける
- 1 論文に対して複数ノートを紐付けできる

### 6.5 PDF 管理

- 外部アプリで開く
- アプリで管理対象ディレクトリを設定できる
- 管理対象ディレクトリ内でフォルダ生成とフォルダ分けを行える
- 論文 PDF を命名規則に従ってリネームできる
- 論文 PDF を所定のフォルダへ移動または整理できる
- ファイル名の命名規則は `{year}_{first_author}_{journal}.pdf` を基本形式とする
- ファイル名に使えない文字は除去または `_` に置換する
- Cloud やドライブで共有しやすいディレクトリ構成を前提とする

### 6.6 分類・整理

- タグ
- コレクション
- お気に入り
- 読了フラグ
- フォルダ単位での整理方針を保持できる

### 6.7 インポート・エクスポート

- 選択した複数論文から BibTeX 用テンプレートを出力できる
- BibTeX 出力時に物理系論文で使いやすい形式へ整形できる
- 出力対象の論文をチェックボックスで選択できる
- メタデータをエクスポートできる
- PDF とメタデータのバックアップ方法を用意する

## 7. 非機能要件

### 7.1 パフォーマンス

- 起動時間は日常利用でストレスのない範囲に収める
- 論文一覧表示と検索は実用上十分な速度で動作する
- 数百件から数千件規模の論文データを扱えることを目標とする

### 7.2 信頼性

- 登録した論文メタデータ、ノート紐付け、PDF パスが再起動後も保持される
- 保存処理中に異常終了しても、データ破損を最小限に抑える
- PDF ファイル移動やリネーム時に失敗した場合、ユーザーが状態を把握できる

### 7.3 セキュリティ

- 個人利用を前提とし、データはローカル環境に保存する
- 外部 API を利用する場合、必要最小限の情報のみ送信する

### 7.4 保守性

- フロントエンド、Tauri、データ管理処理の責務を分離する
- 障害調査のため最低限のログを出力できる
- 主要なデータ操作処理はテスト可能な構成にする

## 11. 技術仕様

### 11.1 アーキテクチャ

- Tauri を用いたデスクトップアプリ
- フロントエンドは React を採用する
- UI 実装、一覧表示、検索、入力フォーム管理は React 側で担う
- ファイル操作、ローカルディレクトリ操作、永続化処理は Tauri / Rust 側で担う
- 永続化方式は JSON ファイルベースとする

### 11.2 ディレクトリ方針

- フロントエンドコード
- Tauri 設定
- 永続化層
- テストコード
- 仕様書管理

### 11.3 永続化方針

- 論文メタデータ、ノート情報、アプリ設定は JSON ファイルとして保存する
- JSON ファイルはアプリを保存しているディレクトリ配下の `setting/` ディレクトリに配置する
- PDF は JSON とは分離してファイルとして保存し、JSON 側ではパスを管理する
- ノートファイルは Markdown などの実ファイルとして保存し、JSON 側では関連パスを管理する
- バックアップは JSON ファイルと PDF、ノートファイルを含むディレクトリ単位で行える構成とする
